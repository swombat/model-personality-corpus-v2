# Corpus Extraction Publishability Review

Date: 2026-05-03

Reviewer: Codex

Scope: verify that `swombat/model-personality-corpus-v2` was extracted correctly from `contemplative-essayist-probe-v2`, and identify fixes needed before making a Zenodo DOI release.

## Summary

The raw data extraction is mostly correct: all shared tracked data, script, and table files in this corpus checkout are byte-identical to the source repository. The corpus repository correctly excludes the papers, review notes, local logs, `keys.env`, `.DS_Store`, and build artifacts, and adds corpus-specific `CITATION.cff` metadata.

I would not publish the current state to Zenodo yet. The main issue is not missing raw traces; it is internal inconsistency between the packaged JSON corpus, the derived tables, and the current analysis scripts.

## Blocking Issues Before Zenodo

### 1. `tables/summary.md` and `tables/cells.tsv` are stale

Running:

```bash
python3 scripts/run_analysis.py
```

rewrites `tables/summary.md` and `tables/cells.tsv`.

The currently committed `tables/summary.md` reports:

- 86 total cells
- 85 cells with at least 1 valid sample
- 2,477 valid freeflow samples
- bin distribution: in=76, transitional=6, out=3, no-data=1

Regenerating from the packaged JSONs reports:

- 149 total cells
- 147 cells with at least 1 valid sample
- 10,063 valid freeflow samples
- bin distribution: in=137, transitional=6, out=4, no-data=2

This is the highest-priority fix. Either regenerate and commit these files, or remove derived freeflow analysis tables from the corpus release if the repository is intended to be raw-data-only.

### 2. `scripts/analyze_per_provider.py` does not reproduce committed `tables/per_provider_*`

Running:

```bash
python3 scripts/analyze_per_provider.py
```

rewrites:

- `tables/per_provider_routing.md`
- `tables/per_provider_routing.tsv`
- `tables/per_provider_pairs.tsv`

The regenerated output removes all `values` rows/sections, while the committed tables include both `freeflow` and `values` analyses.

This means one of the following is true:

- the committed per-provider tables were generated by an older/different script,
- the current script lost the values-analysis path,
- or the values rows should no longer be included in `tables/per_provider_*`.

Before Zenodo, make the policy explicit and reproducible. If values per-provider tables are part of the release, include the script that regenerates them or restore the relevant behavior. If not, regenerate and commit the freeflow-only tables and update the README accordingly.

### 3. `tables_values/` from the source repository is absent

The source repository has:

- `tables_values/per_provider_pairs.tsv`
- `tables_values/per_provider_routing.md`
- `tables_values/per_provider_routing.tsv`

The corpus repository does not include `tables_values/`.

If the values/cross-probe replication tables are intended to be part of this DOI, this is an omission. If the corpus release is intended to be raw traces plus only canonical freeflow tables, then the README should not imply that all values-side derived tables are present.

### 4. README sample schema does not match actual JSON

The README representative sample includes:

```json
"label": "glm-4-6-or-pin-siliconflow",
"or_provider": "siliconflow"
```

Actual trace JSONs do not have top-level `label` or `or_provider` fields. The route/cell label is encoded in the directory name, and OpenRouter's returned upstream provider usually appears as `raw.provider`.

Fix the README sample schema to match the actual files. Suggested wording:

- `cell label` is the containing directory name.
- For OpenRouter responses, the returned upstream may be available as `raw.provider`.
- Pinned route intent is represented by the `*-or-pin-<provider>` cell label and by the collection script's `OR_PROVIDER` / `provider.only` behavior, not by a top-level `or_provider` JSON field.

### 5. `scripts/substrate_rubric.md` contains a local absolute path

`scripts/substrate_rubric.md` says:

```text
/Users/danieltenner/dev/contemplative-essayist-probe-v2/data/traces_freeflow/freeflow_<cellname>/
```

Replace this with a repo-relative path:

```text
data/traces_freeflow/freeflow_<cellname>/
```

This is not a data correctness issue, but it is untidy in a DOI archive.

## Non-Blocking But Worth Deciding

### Raw provider payloads

v2 publishes full provider `raw` payloads in many JSON files. This includes provider response IDs, costs, reasoning fields, occasional request IDs, and some opaque/encrypted-looking nested data blobs in newer responses.

I checked v1 (`swombat/model-personality-probe@v1.0`, linked from Zenodo DOI `10.5281/zenodo.19512754`) and found that v1 also published raw provider payloads:

- 3,960 v1 trace JSON files.
- 1,015 had top-level `raw` payloads.
- 580 OpenRouter samples with raw payloads.
- 435 OpenAI samples with raw payloads.

So keeping raw payloads in v2 is consistent with v1 precedent. v2 raw payloads are richer than v1, but this does not look like a blocker if the authors are comfortable with it.

Recommendation: keep raw payloads, but add a short README note that trace files preserve provider raw response metadata for auditability, including provider IDs/usage/cost/reasoning fields where returned by APIs.

### Exact duplicate result texts

I found 32 groups of exact duplicate non-empty `result` strings, involving 244 files. The most visible cluster is `glm-4-7-or-pin-dekallm`, where repeated outputs occur across multiple indices within the same condition.

This may be real provider/model behavior rather than an extraction bug. I did not find JSON structural errors or duplicate filenames. No action is required unless the papers rely on independence assumptions for those cells; if they do, consider documenting this as provider-side determinism/repetition.

### Partial and zero-valid cells are documented

Partial cells and zero-valid placeholders are present and mostly documented. Examples:

- `deepseek-v4-pro-or-pin-deepseek`: 0 valid freeflow, 0 valid values.
- Fireworks-routed cells: present on disk but excluded from per-provider analysis.
- `glm-5-1-or-pin-fireworks`: 3 valid freeflow, 2 valid values.

README already documents these reasonably well. After the table fixes, re-check that the documented thresholds and exclusions still match generated outputs.

## Recommended Pre-Zenodo Fix Order

1. Decide whether this repository is raw-corpus-only or corpus-plus-derived-analysis.
2. Regenerate or remove stale `tables/summary.md` and `tables/cells.tsv`.
3. Make `tables/per_provider_*` reproducible:
   - restore/generate values sections, or
   - make them explicitly freeflow-only.
4. Decide whether to include `tables_values/`.
5. Fix the README sample schema (`label` / `or_provider` issue).
6. Replace the local absolute path in `scripts/substrate_rubric.md`.
7. Add a short README note that raw provider payloads are intentionally preserved for provenance/auditability.
8. Re-run:

```bash
python3 scripts/corpus_summary.py
python3 scripts/run_analysis.py
python3 scripts/analyze_per_provider.py
git diff --exit-code
```

If `git diff --exit-code` is clean after the intended generated files are committed, the corpus should be in much better Zenodo shape.
