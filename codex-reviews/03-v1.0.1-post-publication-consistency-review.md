# Codex Review 03 — v1.0.1 Post-Publication Consistency Check

Date: 2026-05-04

Scope: review the rushed v1.0.1 corpus update after the MiniMax M2 top-up
and r2 replication cells were added. Focus is internal consistency between
raw traces, generated summaries/tables, README/release metadata, and the
companion paper repo at `~/dev/contemplative-essayist-probe-v2/`.

## Summary

The raw v1.0.1 trace data and the regenerated freeflow/per-provider tables
mostly agree. The new M2 replication cells are present, full, and provider
pinning appears honoured.

The main remaining issue is stale prose/generated summary metadata:
`data/CORPUS_SUMMARY.md`, the corpus `README.md`, and several companion
paper passages still report the pre-v1.0.1 corpus totals. Some M2 headline
prose also still describes the original four-cell M2 comparison rather than
the now-expanded six-cell canonical M2 table.

## Findings

Severity: high for DOI/release consistency.

Current `data/CORPUS_SUMMARY.md` says:

- Freeflow: 10,063 valid samples across 149 cells, 65 cells >=50 valid.
- Values: 8,988 valid samples across 77 cells, 75 cells >=50 valid.
- Combined: 19,051 valid samples across 226 cells.

Raw filesystem count after v1.0.1:

- Freeflow dirs: 151.
- Freeflow non-empty cells: 149.
- Freeflow valid samples: 10,345.
- Freeflow cells >=50 valid: 67.
- Freeflow zero-data dirs: `freeflow_deepseek-v4-pro-or`,
  `freeflow_deepseek-v4-pro-or-pin-deepseek`.
- Values dirs: 77.
- Values non-empty cells: 76.
- Values valid samples: 8,988.
- Values cells >=50 valid: 75.
- Values zero-data dir: `deepseek-v4-pro-or-pin-deepseek`.
- Combined valid samples: 19,333.
- Combined dirs: 228.
- Combined non-empty cells: 225.
- Distinct models: 49.

`tables/summary.md` agrees with the raw freeflow data:

- Total cells: 151.
- Cells with >=1 valid sample: 149.
- Total valid samples: 10,345.

Likely fix: rerun `python3 scripts/corpus_summary.py` and commit the
regenerated `data/CORPUS_SUMMARY.md`. Check whether public-facing prose
should describe "dirs/cell directories" versus "non-empty cells"; the
current generated script reports all directories as cells.

### 2. `README.md` top-line totals and DOI placeholder are stale

- Citation URL `https://doi.org/[DOI to be assigned]`.
- 19,051 valid samples across 226 cells.

all cell directories or only non-empty cells. If following the regenerated

- 228 total cell directories, or 225 non-empty cells if using that definition.
- Freeflow 10,345 valid samples across 151 directories / 149 non-empty cells.

### 3. M2 README prose describes the old four-cell canonical table

Severity: medium.

The current README M2 paragraph says two model-level findings survive
correction and describes three Bonferroni-surviving M2 pairwise comparisons:

- google vs novita.
- google vs minimax-self.
- atlascloud vs google.

After v1.0.1, the canonical `tables/per_provider_routing.tsv` includes six
M2 rows:

- `atlascloud`, n=125, per-25 42.0.
- `google`, n=125, per-25 100.8.
- `google-r2`, n=125, per-25 123.6.
- `minimax`, n=125, per-25 30.4.
- `minimax-r2`, n=125, per-25 29.6.
- `novita`, n=125, per-25 28.0.

The canonical `tables/per_provider_pairs.tsv` now has eight
Bonferroni-significant M2 comparisons:

- atlascloud vs google: d=-0.569, p_Bonf=0.0001824.
- atlascloud vs google-r2: d=-0.616, p_Bonf=4.1e-05.
- google vs minimax: d=0.680, p_Bonf=3.512e-06.
- google vs minimax-r2: d=0.716, p_Bonf=1.045e-06.
- google vs novita: d=0.754, p_Bonf=2.868e-07.
- google-r2 vs minimax: d=0.703, p_Bonf=1.769e-06.
- google-r2 vs minimax-r2: d=0.726, p_Bonf=8.282e-07.
- google-r2 vs novita: d=0.751, p_Bonf=3.56e-07.

Recommended prose shape: distinguish the original four-cell M2 result from
the expanded six-cell v1.0.1 canonical table. That avoids making the original
paper claim look inconsistent while still accurately describing the new
replication cells.

## Checks That Passed

The new M2 replication cells look internally sound:

- `data/traces_freeflow/freeflow_minimax-m2-or-pin-google-r2/`
  - 125 valid samples.
  - `raw.provider == "Google"` for all valid samples.
  - `model == "minimax/minimax-m2"` and
    `model_requested == "minimax/minimax-m2"` for all valid samples.
- `data/traces_freeflow/freeflow_minimax-m2-or-pin-minimax-r2/`
  - 125 valid samples.
  - `raw.provider == "Minimax"` for all valid samples.
  - `model == "minimax/minimax-m2"` and
    `model_requested == "minimax/minimax-m2"` for all valid samples.

No JSON parse errors were found in either probe during the count pass.

`tables/summary.md`, `tables/cells.tsv`, `tables/per_provider_routing.tsv`,
and `tables/per_provider_pairs.tsv` appear regenerated against the new raw
data. The inconsistency is mainly in summary/prose metadata, not in the raw
traces or the current derived tables.
